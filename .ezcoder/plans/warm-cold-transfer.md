# Live Warm/Cold Call Transfer (AI → Human Closer)

## Goal
Let an AI voice agent hand an active Telnyx call to a human closer mid-conversation via a
new `transfer_call` tool. **Warm** mode dials the human, speaks a 1–2 sentence briefing
(caller name, intent, key facts) to the human only, then bridges. **Cold** mode bridges
immediately. Destination is configured per-agent (fallback per-workspace). The handoff is
logged consistently with existing call-event logging and respects the HITL approval gate.

## Verified facts (from research)
- Telnyx Call Control transfer endpoint: `POST /calls/{call_control_id}/actions/transfer`,
  required body `{"to": ...}`; optional `from`, `from_display_name`, `client_state` (base64),
  `custom_headers`, `timeout_secs`, `audio_url` (played to destination before bridging).
- Telnyx also exposes `POST /calls/{ccid}/actions/speak` (`payload`,`voice` required;
  `voice` format `<Provider>.<Model>.<VoiceId>`, e.g. `Telnyx.KokoroTTS.af`) and
  `POST /calls/{ccid}/actions/bridge` (`call_control_id` of the other leg required).
- Voice webhook event types already used: `call.initiated`, `call.answered`, `call.hangup`,
  `call.machine.detection.ended`. Additional real events: `call.bridged`, `call.speak.ended`.
- Voice webhooks dispatched in `backend/app/api/webhooks/telnyx.py` via `_VOICE_HANDLERS`
  keyed on event_type; handlers live in `telnyx_call_handlers.py`. Webhook path is
  `{api_base}/webhooks/telnyx/voice`. `client_state` is base64; helper `encode_client_state`.
- Tools are emitted by `voice_tools.get_tools_from_agent_config()` → `build_tools_list()`
  (OpenAI Realtime: `{"type":"function","name",...}`). Grok builds its own list in
  `grok/session_config.py`; ElevenLabs in `elevenlabs_voice_agent.py`. DTMF is the precedent
  for a call-control tool gated through `tool_settings["call_control"]`.
- Tool execution: `tool_executor.VoiceToolExecutor.execute()` dispatches by name; voice
  callback runs through `approval_gate_service.check_and_execute_or_queue` (unknown action
  types auto-approve unless a HumanProfile policy says `ask`/`never`). `send_dtmf` shows the
  exact pattern for calling `TelnyxVoiceService` from the executor with `call_control_id`.
- Agent model `backend/app/models/agent.py`; schema `backend/app/schemas/agent.py`
  (Create/Update/Response); service `agent_service.py` (`create_agent` uses `**model_dump()`,
  `update_agent` uses `model_dump(exclude_unset=True)`); frontend mirror types
  `frontend/src/types/agent.ts`, request types `frontend/src/lib/api/agents.ts`, form schema
  `frontend/src/lib/agents/agent-form.ts`, edit UI `tabs/advanced-tab.tsx`.
- Frontend tools registry already declares a `call_control` → `transfer_call` tool
  (`frontend/src/lib/integrations.ts`), so enabling it in the Tools tab already writes
  `tool_settings.call_control += "transfer_call"`. We add destination config + execution.
- Single Alembic head: `20260601_drop_contact_tags`. Migrations use guarded DDL pattern.
- Contact PK is `BigInteger`; Agent/Message/Conversation PKs are UUID.
- Verification targets: `make ci.backend`, `make ci.frontend`, `make ci.codegen`
  (commits `backend/openapi.json` + `frontend/src/lib/api/_generated.ts`), `make ci.migrations`.
  `openapi.json` regenerated by `uv run --project backend export-openapi`.

## Design decisions
- **Transfer mechanism (warm):** dial the human as a *new* transfer leg via the Telnyx
  `transfer` action with `client_state` carrying our transfer marker. We DO NOT pass
  `audio_url` (no object storage to host TTS). Instead, after the destination answers we use
  the existing call-control `speak` action targeted at the destination leg, then `bridge`.
  But Telnyx `transfer` auto-bridges the original leg to the destination once answered, so a
  pure-`transfer` approach cannot inject a destination-only briefing cleanly. **Chosen approach:**
  - **Cold:** single `transfer` call (`to`=destination, `from`=workspace number, `client_state`
    marks `cold`). Telnyx bridges automatically on answer. Simple + reliable.
  - **Warm:** use `transfer` with `from_display_name` set to a short caller label AND embed the
    briefing as a leading `speak` is not possible on the auto-bridged leg. So warm uses the
    Telnyx-native pattern: call `transfer` with the briefing delivered via `speak` BEFORE
    transfer to the *caller* ("Connecting you to a specialist now"), and pass the human-facing
    briefing using `from_display_name` (caller id name) + a `custom_headers`/SIP `User-to-User`
    is not audible. Audible destination-only briefing requires a separate dial+speak+bridge.
  - **Final chosen warm implementation (audible, destination-only briefing, no storage):**
    1. AI leg (caller) is put on hold implicitly while we set up. We call
       `speak` on the caller leg: "One moment, connecting you to <name>." (reassurance).
    2. Issue `transfer` to the destination with `client_state` = base64 of a JSON marker
       `{tribunal_transfer: {mode, briefing, message_id, agent_id}}` and
       `webhook_url` defaulted to our voice webhook so `call.answered` for the new leg routes
       back to us. Set `timeout_secs` from config.
    3. On the destination leg's `call.answered` (detected via our transfer marker in
       `client_state`), if mode==warm: call `speak` on the **destination leg** with the
       briefing text (the caller is on the original leg and does not hear it because the legs
       are not yet bridged — Telnyx transfer holds the A-leg until the B-leg answers, and the
       speak targets only the new leg via its own call_control_id), then on `call.speak.ended`
       issue `bridge` of the two legs. For cold mode, Telnyx auto-bridges; we just log.

  > Note on Telnyx semantics: `transfer` creates a new outbound leg and, on answer, bridges
  > it to the original leg. To interpose a destination-only briefing we must own the new
  > leg's `call.answered`/`call.speak.ended` and bridge explicitly. We therefore set
  > `client_state` on the transfer so the new leg's webhooks carry our marker, and we record
  > the original (A) leg's `call_control_id` in the marker so the bridge step can pair them.
  > If precise leg-hold timing proves provider-dependent, the implementation degrades safely:
  > warm falls back to speaking the briefing then a normal transfer (still audible to human,
  > caller hears hold). This is encapsulated in `TelnyxVoiceService.transfer_call()` +
  > the new webhook handlers so the tool contract is stable.

- **Tool name:** `transfer_call` (matches the existing frontend registry + CLAUDE.md anchor).
- **Approval/HITL:** keep intact. The voice tool callback already routes through the approval
  gate; `transfer_call` flows through unchanged (auto unless operator policy says ask/never).
- **Destination resolution:** per-agent `transfer_destination_number` first, else
  `Workspace.settings["transfer_destination_number"]`. If neither set → tool returns a
  friendly error and the AI continues the call.
- **Audit/logging:** structured logs (`transfer_initiated`, `transfer_bridged`,
  `transfer_failed`) consistent with existing `dtmf_sent_via_tool`; persist a marker on the
  call Message via a new `transfer_outcome`/`transfer_to` column set; also write a
  `CallOutcome.signals` style note is out of scope (CallOutcome is created at hangup). We add
  columns on `messages` to record the handoff and surface it in transcripts/logs.

## Risks / mitigations
- **Telnyx warm-leg timing** (caller hearing the briefing): encapsulated in service +
  handlers; warm degrades to "speak reassurance to caller → transfer" if explicit bridging is
  unavailable. Destination still hears the briefing via destination-leg `speak`.
- **client_state collision** with the idempotency UUID base64: marker decoder validates JSON
  shape + a `tribunal_transfer` key and fails closed, so normal calls are unaffected.
- **Approval gate**: unknown action types auto-approve unless policy set; behavior preserved.
- **Multi-provider**: gating helper shared across OpenAI/Grok/ElevenLabs tool builders.
- **Coverage floor**: add focused unit tests for new service + executor + handler branches.
